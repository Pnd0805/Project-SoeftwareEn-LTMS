# สัปดาห์ที่ 2 — ข้อเสนอแนะจากอาจารย์

> แหล่งข้อมูล: ถอดความจากบทสนทนาภาษาไทย โดยเรียบเรียงและย่อให้เข้าใจง่าย ไม่ใช่คำพูดแบบคำต่อคำ

## บทสนทนาที่เรียบเรียงใหม่

### 1. ข้อมูลการแข่งขันและอำนาจของผู้จัดการแข่งขัน

**กลุ่ม:** ระบบแสดงสถานที่แข่งขันและกดเปิดแผนที่ได้ ผู้จัดการแข่งขันสามารถประกาศข่าว เช่น ตารางรอบรองชนะเลิศ ผู้ใช้ดูสายการแข่งขัน รายชื่อทีม โปรไฟล์นักกีฬา รีวิว และส่ง feedback แบบส่วนตัวให้ผู้จัดได้ครับ

**อาจารย์:** ใครสามารถเป็นผู้จัดการแข่งขันได้บ้าง?

**กลุ่ม:** นิสิตสามารถสร้างการแข่งขันเองได้ ส่วนผู้จัดจากภายนอกต้องติดต่อเพื่อรับบัญชี organizer ครับ

**อาจารย์:** ถ้าทุกคนสร้างการแข่งขันได้ ระบบจะสร้างความน่าเชื่อถืออย่างไร? ในระดับมหาวิทยาลัยมีผู้ใช้จำนวนมาก หากทุกคนโพสต์รายการได้อย่างเสรี ระบบจะคล้าย social application และผู้ใช้จะไม่รู้ว่ารายการใดน่าเชื่อถือหรือเหมาะกับตนเอง

**กลุ่ม:** เราตั้งใจให้ใช้การค้นหาและ filter ครับ

**อาจารย์:** แค่นั้นยังไม่พอ ควรมีขั้นตอนการอนุมัติหรือรับรอง เช่น ระบุชมรม คณะ คณะกรรมการกีฬา หรือหน่วยงานมหาวิทยาลัยที่รับผิดชอบรายการนั้นอย่างชัดเจน

### 2. Team manager และวงจรชีวิตของทีม

**อาจารย์:** Team manager คือใคร และทำอะไรได้บ้าง?

**กลุ่ม:** Team manager เป็นหัวหน้าทีมที่สร้างทีม สมัครเข้าร่วมการแข่งขัน และเชิญสมาชิกเข้าทีมได้ สมาชิกกด accept คำเชิญได้ครับ

**อาจารย์:** ทีมเป็นทีมถาวร หรือสร้างขึ้นเฉพาะแต่ละรายการแข่งขัน?

**กลุ่ม:** เป็นทีมชั่วคราวสำหรับแต่ละ tournament ครับ สถิติของทีมเก็บแยกตาม tournament แต่สถิติและประวัติของนักกีฬาจะเก็บในโปรไฟล์ส่วนบุคคล

**อาจารย์:** ขั้นตอนยังไม่ชัด ต้องตัดสินใจให้แน่นอนว่าสร้างทีมก่อนสมัคร หรือสมัครก่อนจึงสร้างทีม รวมถึงกำหนดกรณีสมาชิกไม่ครบ และวิธีทำความเข้าใจประวัติทีมข้าม tournament

### 3. กฎและสิทธิ์การใช้งาน

**อาจารย์:** ต้องกำหนด rule base ให้ชัดเจน คนหนึ่งอยู่ได้กี่ทีม? อยู่หลายทีมใน tournament เดียวกันได้หรือไม่? ถ้าช่วงเวลาแข่งขันไม่ซ้อนกันจะทำได้ไหม? Team manager สร้างหรือส่งได้กี่ทีม?

**กลุ่ม:** คนหนึ่งไม่ควรอยู่หลายทีมใน tournament เดียวกัน แต่สามารถอยู่ทีมจากคนละ tournament ได้ หากเวลาแข่งไม่ซ้อนกันครับ

**อาจารย์:** ต้องบันทึกเป็นกฎของระบบอย่างชัดเจน ไม่ใช่อธิบายด้วยปากเปล่าเฉพาะตอนนำเสนอ

### 4. ผลการแข่งขันและกรรมการ

**อาจารย์:** Organizer ทำอะไรหลังสร้าง tournament แล้ว?

**กลุ่ม:** Organizer กำหนดรายละเอียด tournament เชิญกรรมการ และเผยแพร่รายการแข่งขัน ทีมที่ชนะกรอกผล แล้วกรรมการเข้ามายืนยันผลครับ

**อาจารย์:** ใครเป็นกรรมการได้? ต้องมีกี่คน? กรรมการแต่ละคนตรวจคู่แข่งใดได้บ้าง? หากให้ทีมชนะกรอกผล จะป้องกันคะแนนสดที่ล่าช้าหรือไม่ถูกต้องอย่างไร? ต้องวิเคราะห์ workflow ของการบันทึกและอนุมัติผลให้ครบ

### 5. การทำงานร่วมกันและคุณค่าต่อสมาชิก

**อาจารย์:** สมาชิกทีมทำอะไรได้บ้าง นอกจากกดรับคำเชิญและดูข้อมูล?

**กลุ่ม:** สมาชิกดูข้อมูลและสถิติ เข้าร่วมทีมหรือออกจากทีมได้ครับ

**อาจารย์:** การมีส่วนร่วมยังน้อย ควรเพิ่ม collaboration ระหว่าง organizer กับ team manager เช่น การจัดการทีมไม่มา การสื่อสารกำหนดการ คำขอยกเว้น หรือการจัดการเงื่อนไขการแข่งขัน

### 6. ทำ specification ก่อนขยาย UI

**กลุ่ม:** เราวางแผนให้มีระบบทายผลและสถิติการแข่งขันด้วยครับ

**อาจารย์:** อย่าใช้เวลากับ UI มากเกินไปก่อนกำหนดพฤติกรรมของระบบ ให้ระบุข้อมูล ความแตกต่างของแต่ละกีฬา บทบาท และ workflow ก่อน แล้วใช้ specification เป็น checklist เพื่อตรวจ UI และเขียน user story จากสิ่งนั้น

## ประเด็นที่ต้องแก้ไข

| รหัส | ประเด็น | คำอธิบายและสิ่งที่ต้องแก้ |
| --- | --- | --- |
| W2-01 | การสร้าง tournament ที่ขาดความน่าเชื่อถือ | ไม่ควรให้ผู้ใช้ทุกคนเผยแพร่ tournament ด้วยสิทธิ์เท่ากัน กำหนดประเภท organizer และ workflow การอนุมัติ/รับรอง พร้อมแสดงหน่วยงานผู้สนับสนุนของแต่ละรายการ |
| W2-02 | ขอบเขตของ organizer ไม่ชัดเจน | ระบุว่าใครสร้าง แก้ไข เผยแพร่ ยกเลิก และจัดการ tournament ได้ รวมถึงความแตกต่างระหว่าง organizer ที่เป็นนิสิต บุคคลภายนอก และผู้จัดที่ได้รับอนุญาตจากมหาวิทยาลัย |
| W2-03 | การค้นหารายการยังไม่เพียงพอ | Search และ filter ไม่เพียงพอสำหรับผู้ใช้จำนวนมาก ควรกำหนดหมวดหมู่ คุณสมบัติผู้มีสิทธิ์เข้าร่วม สัญลักษณ์ผู้จัดที่ยืนยันแล้ว ระบบแนะนำ และกฎการมองเห็นรายการ |
| W2-04 | ขั้นตอนทีมขัดแย้งกัน | เลือก workflow เดียวให้ชัด: สร้างทีมแล้วสมัคร หรือสมัครแล้วจึงสร้างทีม พร้อมระบุสถานะทีมไม่ครบ สมัครไม่ผ่าน ถอนตัว และจบการแข่งขัน |
| W2-05 | ผลกระทบของทีมชั่วคราว | อธิบายความสัมพันธ์ระหว่างทีมชั่วคราวกับประวัติระยะยาว เก็บประวัตินักกีฬาให้ต่อเนื่อง และตัดสินใจว่าจำเป็นต้องมีตัวตนทีมแบบถาวรหรือไม่ |
| W2-06 | ขาดกฎเกี่ยวกับทีม | กำหนดการเลือก manager จำนวนสมาชิก ตัวจริง/ตัวสำรอง การเชิญ การตอบรับ กำหนดเวลาตอบรับ การออกจากทีม และจำนวนทีมที่ผู้ใช้หรือ manager สร้าง/เข้าร่วมได้ |
| W2-07 | การอยู่หลายทีมไม่ชัดเจน | กำหนดให้ชัดว่าผู้เล่นอยู่ได้หลายทีมหรือไม่ โดยเฉพาะใน tournament เดียวกัน หากอนุญาตข้าม tournament ตามเวลาที่ไม่ทับซ้อน ต้องระบุ logic การตรวจสอบ |
| W2-08 | Workflow ของกรรมการไม่ครบ | ระบุวิธีแต่งตั้ง จำนวนกรรมการ การมอบหมายคู่แข่ง สิทธิ์ในการอนุมัติ และการจัดการเมื่อกรรมการไม่ว่างหรือมีข้อโต้แย้งในผลการแข่งขัน |
| W2-09 | ความเสี่ยงจากการให้ทีมชนะกรอกผล | ไม่ควรพึ่งพาทีมชนะเพียงฝ่ายเดียว กำหนดผู้กรอกคะแนน หลักฐาน/การยืนยัน การตรวจโดยกรรมการ การแก้ไขผล และการแสดงสถานะคะแนนชั่วคราวเทียบกับคะแนนยืนยันแล้ว |
| W2-10 | Collaboration ไม่เพียงพอ | เพิ่มการทำงานที่รองรับการจัดงานจริง เช่น ประกาศ การสื่อสาร organizer–manager การเช็กอิน/ไม่มาตามนัด การเปลี่ยนกำหนดการ คำขอ และการจัดการข้อพิพาท |
| W2-11 | ข้อมูลและสถิติยังไม่ถูกกำหนด | ระบุทุกสถิติ แหล่งข้อมูล การแยกตามชนิดกีฬา ระดับบุคคล/ทีม/tournament และการบันทึกตัวจริง/ตัวสำรอง หลีกเลี่ยงการรวมสถิติที่ต่างชนิดกีฬากันโดยไม่มีความหมาย |
| W2-12 | ทำ UI นำ specification | จัดทำ functional specification, role-permission matrix, rule base, state flow และ user story ก่อนขยาย mock-up แล้วใช้เอกสารเหล่านี้ตรวจสอบทุกหน้าจอ |
| W2-13 | ขอบเขตระบบและ master data ไม่ชัด | ตัดสินใจว่าระบบเป็นระดับมหาวิทยาลัยหรือไม่ รองรับกีฬา หมวดหมู่ tournament คณะ และหน่วยงานใดบ้าง พร้อมกำหนด master data และความต่อเนื่องของประวัติ/การติดตาม เพื่อไม่ให้ระบบกลายเป็น social app ทั่วไป |
| W2-14 | การตรวจคุณสมบัติยังไม่ละเอียด | ระบุว่าเงื่อนไขอายุ คณะ สัดส่วนเพศ จำนวนสมาชิก ประสบการณ์ และผลงาน เป็น hard หรือ soft rule รวมถึงกำหนดการตรวจทั้งก่อนสมัคร หลังรับเข้าทีม และตอนส่งใบสมัคร และพิจารณาว่าทีมต้องได้รับการรับรองจากคณะหรือมหาวิทยาลัยหรือไม่ |
| W2-15 | การยืนยันตัวตนและสิทธิ์ของ Outsider ไม่ชัด | กำหนดวิธีตรวจตัวตน อายุ และคุณสมบัติของบุคคลภายนอก รวมถึง permission จากมหาวิทยาลัยก่อนสมัครหรือแข่งขัน และแยกสิทธิ์การดู การทายผล การสมัคร และการเข้าร่วมแข่งขันให้ชัด |
| W2-16 | หน้าที่ของเจ้าหน้าที่มหาวิทยาลัยยังไม่ชัด | ระบุว่าเจ้าหน้าที่ตรวจ อนุมัติ ปฏิเสธ หรือบันทึกอะไรในแต่ละช่วงของ tournament รวมถึงการรับรอง organizer การอนุญาต Outsider และการยกระดับปัญหาเมื่อ organizer หรือกรรมการไม่ดำเนินการ |
| W2-17 | การเปลี่ยนบทบาทเป็น Organizer ไม่ชัด | ระบุว่าการสร้าง tournament ทำให้ได้บทบาท organizer ทันทีหรือเฉพาะหลังอนุมัติ บทบาทมีผลเฉพาะ tournament นั้นหรือไม่ และการกระทำใดถูกจำกัดก่อนการยืนยันตัวตน |
| W2-18 | การเชื่อมต่อ live stream ภายนอกยังไม่ชัด | ระบุขอบเขตการเชื่อม YouTube วิธี link/embed ผู้รับผิดชอบ stream ข้อมูลสถานะ live และ replay ที่จะแสดง และพฤติกรรมเมื่อวิดีโอภายนอกใช้งานไม่ได้ |
| W2-19 | การแสดง bracket และการจับคู่ไม่รองรับจำนวนมาก | แสดงเส้นเชื่อมระหว่างรอบ ทีมคู่แข่ง ผู้ชนะ และช่อง TBD ให้ชัด กำหนด manual/random pairing และ layout ที่อ่านได้เมื่อมี 30–50 ทีม รวมถึงรูปแบบการแข่งขัน เช่น knockout |
| W2-20 | สิทธิ์และช่วงเวลาการทายผลไม่ครบ | ระบุว่าใครทายผลได้ ทายได้ถึงเมื่อไร ผู้แข่งขันและ organizer ถูกยกเว้นหรือไม่ ความสัมพันธ์ระหว่างผลทายกับคะแนนชั่วคราว/คะแนนยืนยันแล้ว และการเก็บผลทายระดับ match หรือ tournament |
| W2-21 | ไม่มีรายการหน้าและ user journey ที่ครบ | นับและทำแผนผังหน้า สถานะ และ journey ของแต่ละบทบาท แทนการนำเสนอเฉพาะหน้าที่ mock ไว้ เพื่อค้นหา flow ที่ขาดและตรวจว่า UI อธิบาย specification ได้ครบ |

## เอกสารที่ควรทำต่อไป

1. ตารางบทบาทและสิทธิ์: Student, team member, team manager, organizer, official และ university administrator
2. วงจรชีวิต tournament และ workflow การอนุมัติ
3. วงจรชีวิตทีมและ rule base ของการเข้าร่วม
4. Workflow การรายงานผลและการยืนยันโดยกรรมการ
5. User stories สำหรับการทำงานร่วมกันระหว่าง organizer, manager และสมาชิก
6. Data dictionary สำหรับ tournament, team, athlete, match, score และสถิติเฉพาะกีฬา
7. กฎคุณสมบัติและการยืนยันตัวตนสำหรับนิสิต KU และ Outsider
8. Workflow ของเจ้าหน้าที่มหาวิทยาลัยและ specification การเชื่อม YouTube
9. กฎ bracket/pairing และการทายผล พร้อมสถานะ UI ที่รองรับ tournament ขนาดใหญ่
